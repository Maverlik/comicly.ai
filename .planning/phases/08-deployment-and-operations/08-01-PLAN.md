<objective>
Create the minimal deployment configuration needed for separate Vercel frontend and backend projects.

Purpose: Make production deployment safe by publishing an explicit frontend static output and a standalone `backend/` FastAPI project, without exposing repo-root files or coupling production to Docker Compose.
Output: root frontend Vercel config/build step, backend Vercel config, Python runtime declaration, and deployment config tests.
</objective>

<threat_model>
- **T-08-01 Threat:** Vercel root frontend deploy exposes `.env`, `.planning/`, `backend/`, package metadata, or other private files. **Mitigation:** Add an explicit allow-list static build output and test that private paths are not copied.
- **T-08-02 Threat:** Backend deploy accidentally depends on local Docker Compose or dev server behavior. **Mitigation:** Backend Vercel config runs FastAPI directly and docs keep Docker Compose local-only.
- **T-08-03 Threat:** Synchronous generation exceeds default function duration. **Mitigation:** Configure/document max duration where supported and preserve future async fallback notes.
- **T-08-04 Threat:** Vercel Python Beta constraints leak into app design. **Mitigation:** Keep app code portable and isolate Vercel-specific settings to deployment config/docs.
</threat_model>
