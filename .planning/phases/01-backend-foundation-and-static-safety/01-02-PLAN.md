<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client to health routes | Anonymous clients can call public operational endpoints. |
| app to PostgreSQL | Backend crosses into database dependency for readiness checks. |
| Alembic to database | Migration tooling connects using runtime configuration. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | I | `GET /health` | mitigate | Return only minimal process status; do not expose provider config, model IDs, database URLs, or secret presence. |
| T-02-02 | I | `GET /ready` | mitigate | Map database failures to `DATABASE_UNAVAILABLE` without raw exception details or connection strings. |
| T-02-03 | D | `backend/app/db/health.py` | mitigate | Use a bounded simple `SELECT 1` readiness check through pooled async SQLAlchemy engine. |
| T-02-04 | T | `backend/alembic/env.py` | mitigate | Bind Alembic to app metadata only; do not create or mutate business schema outside explicit migrations. |
</threat_model>
