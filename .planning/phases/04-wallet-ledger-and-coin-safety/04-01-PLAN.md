<task type="auto" tdd="true">
  <name>Task 2: Enforce idempotency and insufficient balance behavior</name>
  <files>backend/app/services/wallets.py, backend/tests/test_wallet_service.py</files>
  <behavior>
    - Billable debits require a non-empty idempotency key.
    - Missing key raises `ApiError` with status `400` and code `IDEMPOTENCY_KEY_REQUIRED`.
    - Reusing the same key for the same logical operation returns the existing transaction/result without another balance change.
    - Insufficient funds raise `ApiError` with status `409` and code `INSUFFICIENT_COINS`.
    - Insufficient funds do not insert a debit transaction.
  </behavior>
  <action>Add idempotency lookup/replay before mutation, server-side balance checks, and clear error mapping.</action>
  <verify>
    <automated>cd backend; python -m pytest tests/test_wallet_service.py -k "idempotency or insufficient"</automated>
  </verify>
  <done>Retries and double clicks cannot double-charge or drive the balance negative.</done>
</task>

<task type="auto" tdd="true">
  <name>Task 3: Add refund and concurrency coverage</name>
  <files>backend/app/services/wallets.py, backend/tests/test_wallet_service.py</files>
  <behavior>
    - Refund primitive creates a positive transaction linked to the failed/debited reference.
    - Duplicate refund idempotency key restores balance only once.
    - Concurrent debit attempts cannot create a negative balance.
    - PostgreSQL path uses row locking or an equivalent atomic strategy; SQLite tests use a deterministic compatible fallback.
  </behavior>
  <action>Implement refund support and race-safety around balance mutation. Keep network generation outside the transaction boundary.</action>
