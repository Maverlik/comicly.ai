# ТЗ по backend: Comicly.ai

## 1. Цель

Нужно реализовать production-ready backend для `comicly.ai`, который:

- авторизует пользователей через Google и Яндекс;
- хранит пользователей, профиль, баланс монет, историю списаний и историю всех созданных комиксов;
- обеспечивает списание монет при генерации;
- готов к будущему подключению оплаты пакетов монет;
- разворачивается в облаке и корректно работает в проде.

Документ составлен с учетом текущего проекта:

- фронт сейчас статический: `index.html`, `create.html`, `scripts/main.js`, `scripts/creator.js`;
- текущий backend минимальный: `server.js`;
- уже есть AI-эндпоинты: `POST /api/generate-comic-page`, `POST /api/ai-text`, `GET /api/health`;
- на фронте пока используются демо-данные профиля и захардкоженные кредиты.

## 2. Общие требования

### 2.1 Что должно получиться

После завершения работ пользователь должен иметь возможность:

- зайти на сайт через Google или Яндекс;
- увидеть свой актуальный баланс монет;
- потратить монеты на генерацию;
- открыть профиль и изменить имя и аватар;
- увидеть в профиле историю всех созданных комиксов;
- чтобы все данные хранились и читались из БД, а не из памяти клиента;
- чтобы сайт был задеплоен и работал в production.

### 2.2 Базовые технические принципы

- Источник истины по монетам только БД.
- Списание монет только на бэкенде.
- Нельзя доверять балансу, который приходит с фронта.
- Любая операция списания/начисления монет должна писать запись в журнал транзакций.
- Все критичные операции должны быть идемпотентны или защищены от повторного выполнения.
- Авторизация и сессии должны быть безопасными: `HttpOnly` cookie, `Secure` в production, `SameSite=Lax` или строже.
- Все приватные маршруты доступны только авторизованному пользователю.

## 4. Функциональные требования

### 4.1 Авторизация через Google и Яндекс

Нужно реализовать OAuth-вход через:

- Google;
- Яндекс ID.

Требования:

- вход и регистрация должны происходить без отдельного пароля;
- если пользователь впервые вошел через провайдера, в БД создается новый пользователь;
- если пользователь уже существует, выполняется вход в существующий аккаунт;
- привязка выполняется по провайдеру и его уникальному `provider_user_id`;
- email, имя и аватар из провайдера можно использовать для первичного заполнения профиля;
- после успешного входа создается серверная сессия;
- должен быть logout.

Дополнительно:

- backend обязан валидировать токены/authorization code на своей стороне;
- нельзя принимать "как есть" идентификатор пользователя, переданный фронтом без серверной проверки;
- redirect URI и домены production/staging должны быть корректно настроены.

### 4.2 Система монет

Нужно реализовать полноценную систему монет.

Требования:

- у каждого пользователя есть баланс монет;
- баланс хранится в БД;
- на фронт баланс приходит только с backend API;
- любое списание и начисление фиксируется в отдельной таблице транзакций;
- баланс должен пересчитываться корректно и не уходить в минус;
- при недостатке монет backend возвращает понятную ошибку `409` или `402/400` с кодом ошибки.

Бизнес-правила для текущей версии:

- генерация полной страницы комикса: `20` монет;
- перегенерация сцены: `4` монеты;
- стоимости должны храниться в конфиге или в отдельной таблице, а не хардкодиться по всему коду;
- стартовый баланс нового пользователя должен задаваться конфигом приложения.

Критично:

- списание монет и запись результата генерации должны быть в одной транзакционной логике;
- нельзя допустить двойное списание из-за двойного клика, повторного запроса или retry;
- если генерация завершилась ошибкой до успешного результата, монеты не должны теряться либо должен быть предусмотрен автоматический refund.

### 4.3 Профиль пользователя

В профиле пользователь должен иметь возможность:

- изменить отображаемое имя;
- загрузить/обновить аватар;
- получить текущие данные профиля.

Требования:

- имя редактируется через API и сохраняется в БД;
- аватар загружается в хранилище, в БД хранится ссылка/ключ файла;
- должны быть проверки типа файла и разумного лимита по размеру;
- старый аватар не должен оставаться "висящим мусором" без необходимости;
- нужно вернуть фронту URL готового аватара.

### 4.4 История комиксов

Нужно сохранять в БД историю всех созданных пользователем комиксов.

Минимально в истории должны храниться:

- id комикса;
- пользователь;
- название проекта;
- исходный сюжет;
- персонажи;
- стиль;
- тон;
- дата создания;
- дата последнего изменения;
- список страниц комикса;
- по каждой странице: номер страницы, изображение/URL, модель генерации, стоимость в монетах, дата генерации;
- статус сущности: `draft`, `generated`, `failed`, `archived` или аналогичный набор.

Требования к API:

- получить список всех комиксов текущего пользователя;
- получить один комикс по id;
- создать новый комикс;
- обновить метаданные комикса;
- сохранить новую сгенерированную страницу в рамках комикса;
- нельзя отдавать пользователю чужие комиксы.

### 4.5 Таблицы под оплату монет

Нужно подготовить БД к будущей оплате монет.

На текущем этапе обязательно:

- создать таблицу продуктов/пакетов монет;
- создать таблицу платежей;
- предусмотреть статусы платежа;
- предусмотреть привязку платежа к пользователю и к пакету монет;
- предусмотреть поля для внешнего платежного провайдера.

Минимальные пакеты:

- `100` монет;
- `500` монет;
- `1000` монет.

Что нужно сейчас:

- миграции таблиц;
- seed базовых пакетов;
- API получения списка пакетов.

Что можно не делать в этой задаче, если отдельно не согласовано:

- реальную интеграцию эквайринга/кассы;
- реальное начисление монет после webhook от платежного провайдера.

Но структура БД должна быть готова к этому без перелома модели данных.

### 4.6 Продакшен-деплой

Нужно развернуть проект так, чтобы он работал как production-сервис.

Допустимые варианты:

- Vercel;
- или аналогичный хостинг, если он лучше подходит под финальную архитектуру.

Требования:

- production URL должен открываться и работать;
- авторизация через Google и Яндекс должна работать именно на production-домене;
- env-переменные должны быть заведены в окружение деплоя;
- БД должна быть подключена к production;
- приватные API должны работать с production cookie/session;
- должны быть настроены CORS/CSRF-поведения в соответствии с финальной схемой доменов;
- должна быть инструкция, как повторить деплой.

## 7. Бизнес-логика генерации

Для генерации страницы процесс должен быть таким:

1. Проверить сессию пользователя.
2. Провалидировать входные данные.
3. Проверить баланс.
4. Создать/обновить сущность комикса.
5. Запустить генерацию.
6. После успеха сохранить страницу комикса.
7. Списать монеты и записать транзакцию.
8. Вернуть URL страницы, данные страницы и новый баланс.

Если генерация упала:

- в БД должен остаться корректный статус ошибки;
- баланс не должен уменьшиться без успешного результата;
- ошибка должна быть логируема и понятна фронту.

## 8. Безопасность и надежность

Обязательно:

- серверная валидация входных данных;
- защита приватных маршрутов;
- безопасная работа с cookie/session;
- ограничения на upload аватаров;
- базовая rate limit-защита для auth и генерации;
- логирование ошибок;
- отсутствие секретов в клиентском коде;
- миграции и возможность поднять проект с нуля.

Для монет обязательно:

- транзакционность;
- защита от race condition;
- защита от повторного списания;
- аудит всех изменений баланса.

## 9. Что нужно изменить в текущем проекте

Минимально нужно убрать демо-логику и перевести ее на реальные данные:

- текущий `credits` на фронте не должен хранить фактический баланс;
- текущий mock-профиль в `create.html` / `scripts/creator.js` должен получать реальные данные пользователя;
- история комиксов не должна жить только в состоянии страницы;
- backend должен перестать быть "только proxy к OpenRouter" и стать системой с БД, auth и business logic.

## 10. Env-переменные

Минимально предусмотреть:

- `DATABASE_URL`
- `SESSION_SECRET`
- `APP_URL`
- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `YANDEX_CLIENT_ID`
- `YANDEX_CLIENT_SECRET`
- `OPENROUTER_API_KEY`
- переменные для object storage
- `STARTER_COINS`

## 11. Критерии приемки

Задача считается выполненной, если:

1. Новый пользователь может зайти через Google.
2. Новый пользователь может зайти через Яндекс.
3. После первого входа пользователь создается в БД.
4. У пользователя есть кошелек в БД.
5. Баланс отдается из БД и совпадает с историей транзакций.
6. При генерации страницы монеты реально списываются на бэкенде.
7. При недостатке монет генерация блокируется корректной ошибкой.
8. Пользователь может изменить имя профиля.
9. Пользователь может загрузить и сменить аватар.
10. В БД сохраняются все созданные комиксы и их страницы.
11. Пользователь может получить список своих комиксов.
12. В БД есть таблицы пакетов монет и платежей.
13. В БД есть минимум 3 пакета: 100 / 500 / 1000.
14. Production deployment поднят и работает.
15. OAuth на production-домене реально работает.
16. Есть инструкция запуска локально и инструкция деплоя.

## 12. Что передать по результату работ

В финале backend-разработчик должен передать:

- код;
- миграции;
- `.env.example`;
- краткий `README` по запуску;
- список env-переменных;
- production URL;
- список реализованных API;
- описание структуры БД.

## 13. Что не входит в эту задачу, если отдельно не согласовано

- полноценная платежная интеграция с конкретным провайдером;
- админка;
- email-рассылки;
- система ролей;
- социальные функции;
- публичные профили;
- публикация комиксов в ленту.

## 14. Важные замечания для реализации

- Не хранить монеты только в поле `balance` без журнала операций.
- Не списывать монеты на фронте "оптимистично" без подтверждения backend.
- Не принимать решения по авторизации на основании данных, пришедших только из браузера.
- Не делать модель БД только "под сегодня": таблицы оплаты должны быть готовы к реальному провайдеру.
- Не ломать уже существующие AI-маршруты без плана миграции фронта.
